## สองประเภทที่ทรงพลังในโมดูลเดียว

โมดูลนี้แนะนำโครงสร้างข้อมูลที่มีประโยชน์ที่สุดสองชนิดใน Redis ได้แก่ **Lists** และ **Hashes**

**List** คือชุดข้อมูลแบบมีลำดับที่ใช้ doubly-linked list เป็นโครงสร้างภายใน คุณสามารถ push และ pop ข้อมูลจากปลายทั้งสองได้ในเวลา O(1) ไม่ว่าจะมีสมาชิกกี่รายการ จึงเหมาะอย่างยิ่งสำหรับคิวงาน สแต็ก ฟีดกิจกรรม และรายการไอเทมล่าสุด

**Hash** คือ map แบบ field-value ที่เก็บอยู่ภายใต้ key เดียว ลองนึกภาพเหมือน `dict` ของ Python หรือ object ของ JavaScript คุณสามารถกำหนดและดึงค่าของแต่ละ field ได้แบบ atomic หมายความว่าสามารถอัปเดตอีเมลของผู้ใช้โดยไม่กระทบข้อมูล field อื่น ๆ เลย

```text
127.0.0.1:6379> RPUSH notifications "login" "purchase" "logout"
(integer) 3
127.0.0.1:6379> LRANGE notifications 0 -1
1) "login"
2) "purchase"
3) "logout"
127.0.0.1:6379> HSET user:1 name "Ada" email "ada@example.com" age "30"
(integer) 3
127.0.0.1:6379> HGETALL user:1
1) "name"
2) "Ada"
3) "email"
4) "ada@example.com"
5) "age"
6) "30"
```

## เลือกใช้ประเภทไหนดี?

| ประเภท | ใช้เมื่อต้องการ… | ตัวอย่าง |
|--------|-----------------|---------|
| **List** | ข้อมูลมีลำดับ เข้าถึงหัวท้ายด้วย O(1) | คิวข้อความ, ฟีดกิจกรรม, คำค้นหาล่าสุด |
| **Hash** | Field มีชื่อภายใต้ key เดียว อัปเดตบางส่วนได้ | โปรไฟล์ผู้ใช้, ข้อมูลสินค้า, ข้อมูล session |

## ลองสัมผัสทั้งสองประเภท

ลองเพิ่มข้อมูลลงใน list และเก็บ field ต่าง ๆ ใน hash:

```text
127.0.0.1:6379> RPUSH tasks "write tests" "deploy" "notify team"
(integer) 3
127.0.0.1:6379> LRANGE tasks 0 -1
1) "write tests"
2) "deploy"
3) "notify team"
127.0.0.1:6379> HSET product:99 name "Widget" price "9.99" stock "100"
(integer) 3
127.0.0.1:6379> HGET product:99 name
"Widget"
127.0.0.1:6379> HGET product:99 price
"9.99"
```
