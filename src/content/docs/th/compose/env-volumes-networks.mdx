## Environment variables ใน Compose

มีสองวิธีในการส่ง environment variables เข้าเซอร์วิส

### Inline environment

กำหนด variables โดยตรงใน compose.yaml:

```yaml
services:
  app:
    image: myapp:latest
    environment:
      - NODE_ENV=production
      - PORT=3000
```

### env_file

อ้างอิงไฟล์ภายนอก Compose อ่านแต่ละบรรทัด `KEY=VALUE` จากไฟล์และฉีดเข้าคอนเทนเนอร์:

```yaml
services:
  app:
    image: myapp:latest
    env_file:
      - .env
```

ไฟล์ `.env`:

```text
NODE_ENV=production
PORT=3000
DATABASE_URL=postgres://db:5432/mydb
```

เก็บ `.env` ออกจาก version control — มักมี secrets อยู่

## Variable interpolation ด้วย `${VAR}`

Compose อ่านไฟล์ `.env` ใน directory เดียวกับ compose.yaml และใช้มันแทนที่ placeholder `${VAR}` ภายใน compose.yaml เอง:

```yaml
services:
  db:
    image: postgres:17-alpine
    environment:
      - POSTGRES_PASSWORD=${DB_PASSWORD}
      - POSTGRES_DB=${DB_NAME}
```

ด้วยไฟล์ `.env` ที่มี:

```text
DB_PASSWORD=supersecret
DB_NAME=mydb
```

Compose แทนที่ค่าก่อนสตาร์ทคอนเทนเนอร์ นี่ทำให้ secrets ไม่ต้องอยู่ใน compose.yaml และให้นักพัฒนาแต่ละคนใช้ไฟล์ `.env` ของตนเองได้

## Named volumes

คอนเทนเนอร์เป็น ephemeral — ข้อมูลที่เขียนภายในคอนเทนเนอร์จะหายเมื่อคอนเทนเนอร์ถูกลบ Named volumes เก็บข้อมูลบน host และอยู่รอดได้แม้คอนเทนเนอร์จะ restart หรือถูก recreate

ประกาศ named volume ที่ระดับบนสุดและ mount ต่อเซอร์วิส:

```yaml
services:
  db:
    image: postgres:17-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=secret

volumes:
  pgdata:
```

key `volumes:` ระดับบนสุด register volume กับ Compose รายการ `volumes:` ต่อเซอร์วิส mount มัน คุณสามารถ mount named volume เดียวกันในหลายเซอร์วิสได้

## Custom networks

Compose สร้าง default network อัตโนมัติ สำหรับการควบคุมมากขึ้น — แยก services หรือเชื่อมต่อ services ข้าม Compose files — กำหนด custom networks:

```yaml
services:
  frontend:
    image: nginx:alpine
    networks:
      - public

  backend:
    image: myapp:latest
    networks:
      - public
      - private

  db:
    image: postgres:17-alpine
    networks:
      - private

networks:
  public:
  private:
```

ที่นี่ `frontend` เข้าถึง `backend` ได้ แต่ `frontend` ไม่สามารถเข้าถึง `db` โดยตรง `backend` เชื่อม network ทั้งสอง

## ลองปฏิบัติจริง

snippet ด้านล่างเชื่อมต่อเซอร์วิสกับ env file และ named volume เพื่อให้ข้อมูลอยู่รอดได้หลัง restart

export const envVolNetCode = `# Create .env file
cat > .env <<'EOF'
APP_MSG=Hello from .env interpolation!
VOLUME_DATA_DIR=/data
EOF

# Create compose.yaml with env interpolation and a named volume
cat > compose.yaml <<'EOF'
services:
  writer:
    image: alpine
    env_file:
      - .env
    volumes:
      - mydata:\${VOLUME_DATA_DIR}
    command: >
      sh -c "
        echo \${APP_MSG} &&
        echo 'Writing to named volume...' &&
        echo 'persistent data' > \${VOLUME_DATA_DIR}/file.txt &&
        echo 'Wrote file to volume.'
      "

  reader:
    image: alpine
    volumes:
      - mydata:/data
    command: >
      sh -c "
        sleep 2 &&
        echo 'Reading from volume:' &&
        cat /data/file.txt
      "
    depends_on:
      - writer

volumes:
  mydata:
EOF

# Run both services
docker compose up

# Verify the volume still exists after containers stop
docker volume ls | grep mydata

# Clean up (keeps the volume)
docker compose down

# Clean up including the volume
docker compose down -v`;
